\begin{abstract}
Reliability is a recurring problem in agentic tasks that require both natural conversation and faithful procedure execution. A model must respond coherently while also obeying ordering constraints, branch conditions, and completion criteria. We study a Tactus-based support-intake benchmark and compare two architectures that conceptually model MCP-style ``elicitation'' without an MCP server: an \textbf{unguided} baseline where the LLM must track and decide when to capture structured fields, and a \textbf{guided} design where the procedure triggers explicit form checkpoints for branch-critical fields and validates/stores responses programmatically. In a 10-run simulated-user study (three personas), guided checkpoints achieve 100\% strict success and 100\% completion, while the unguided baseline completes often but misses strict success (70--80\% strict; 70--90\% completion), with 0\% infra failures in both arms.
\end{abstract}

\section{Introduction}
Support intake flows mix two objectives that can conflict: maintaining a natural conversation and satisfying a strict checklist (disclosures, identifiers, branch-specific requirements, and approvals). When the same language model is responsible for both conversational grounding and procedural bookkeeping, reliability often suffers: steps get skipped, fields are recorded out of order, or the flow ends prematurely.

This repo explores an alternative split inspired by MCP \emph{elicitation}: treat certain structured inputs as \emph{authoritative checkpoints} controlled by orchestration rather than free-form conversational inference.

\section{Method}

\subsection{Task and procedure}
The support-intake task requires a recording/privacy disclosure, issue categorization, collection of core fields, branch-specific steps (billing or technical), explicit plan approval, and a final \texttt{done} decision. Validators and guards enforce ordering constraints; for example, \texttt{account\_email} should not be recorded before the recording/privacy disclosure has been logged.

\subsection{Unguided vs.\ guided elicitation-style checkpoints}
The two procedure variants implement the same overall task:
\begin{itemize}[noitemsep]
  \item \textbf{Unguided baseline:} the LLM decides when and how to capture fields during the conversation.
  \item \textbf{Guided checkpoints:} the procedure triggers explicit ``form'' prompts for branch-critical fields and validates/stores the answers programmatically.
\end{itemize}

\subsection{Simulated user}
\texttt{LLMHITLHandler} replaces \texttt{Human.input} with an LLM role-playing a persona with private ground-truth fields (\texttt{tests/support\_personas.py}). Optional pairing via a deterministic seed can reduce user-side variance across arms.

\subsection{Reliability harness}
For each combination of arm $\times$ persona, pytest runs $N$ independent procedure executions. To avoid asyncio teardown issues under concurrency, each execution runs inside \texttt{asyncio.run} in a worker thread, while pytest's event loop only awaits bounded parallel work guarded by a semaphore. \section{Metrics}
\begin{itemize}[noitemsep]
  \item \textbf{Strict success:} procedure reports completed and all ground-truth fields match.
  \item \textbf{Completion:} procedure reports completed (regardless of field correctness).
  \item \textbf{Infra failure:} execute/API error (excluded from strict ex-infra headline when reported separately).
\end{itemize}
\texttt{scripts/compare\_reliability.py} prints guided vs.\ unguided tables and Wilson 95\% intervals for binomial rates (\texttt{--no-ci} to omit).

\section{Discussion}
In this sample, guided checkpoints eliminate the observed failures in both completion and strict correctness. The unguided baseline is meaningfully \emph{believable} (it often completes) but still fails strict correctness due to missed requirements or incorrect final structured state.

\section{Limitations}
This study uses simulated users rather than real customer transcripts, so measured rates may not transfer directly to production behavior. The reported results are for a small sample size (10 runs per persona per arm) and one guide model family. The strict metric is intentionally brittle: benign paraphrases or alternate-but-plausible captures can count as failure when they diverge from the ground truth or required normalized form.
